# Design Systems — Carbon (IBM)

## Purpose
Reference notes on **Carbon (IBM)** to inform decisions. Study; do not copy.

## What it does well
- Voice: enterprise clarity, dense data-friendly components, disciplined tokens.
- Token discipline is a hallmark; follow its structure, not its literal values.
- Component patterns worth learning: navigation model, form patterns, feedback conventions.

## What to borrow
- Spacing scale discipline.
- State conventions (hover, focus, disabled).
- Motion primitives and reduced-motion practice.
- Component naming clarity.

## What NOT to copy
- Brand voice, palette, or typography wholesale.
- Iconography that carries the system's identity.
- Component ergonomics that assume the parent product's audience.

## When to reference this system
- When the user explicitly names it.
- When the target audience is trained on this system (Apple → iOS-heavy audiences, GitHub → developer audiences, etc.).
- Never as a default just because it exists.

## Anti-patterns
- Combining tokens from multiple systems (creates incoherence).
- Adopting an entire system when a handful of principles would do.
- Presenting a look-alike site as a fresh design.
