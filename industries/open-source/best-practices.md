# Open Source — Best Practices

## UX priorities
- Design for two audiences: evaluators (does this solve my problem?) and contributors (can I get involved?).
- Fastest path to install/quick-start is the highest-value action.
- Documentation is the product surface; treat it as UI.

## Content strategy
- Lead with what the project is and what it replaces or solves — in one sentence.
- Show code within the first fold if the project is developer-facing.
- Community signals (stars, contributors, releases) build initial trust.

## Interaction guidance
- Copy-to-clipboard on install commands is required, not optional.
- Docs search is a critical UI, not a nice-to-have.
- Never require signup or email to try the project.
